# SwarmFed-RL 项目技术详解与面试指南

## 1. 项目背景与目标 (Situation & Task)

**项目名称**：SwarmFed-RL (基于 ROS2 的去中心化群体联邦强化学习系统)

**核心痛点**：
在多机器人协同探索（如搜救、仓储物流）场景中，传统方法面临两大挑战：
1. **通信瓶颈**：依赖中央服务器的传统联邦学习（Centralized FL）在网络不稳定或带宽受限时容易失效，且存在单点故障风险。
2. **样本效率低**：机器人各自独立训练（Local RL）收敛极慢，无法利用群体的经验。
3. **安全威胁**：在开放网络中，恶意节点可能注入脏数据（Poisoning Attack）破坏群体模型。

**解决方案**：
我设计并实现了一套**去中心化（P2P）联邦强化学习框架**。
- 机器人通过**Soft Actor-Critic (SAC)** 算法在本地与环境交互。
- 利用 **ROS2** 通信机制，在物理相遇时触发 **Actor 权重交换**。
- 引入 **拜占庭容错（Byzantine Fault Tolerance）** 机制防御恶意攻击。

## 2. 核心技术架构 (Action - Tech Stack)

### 2.1 强化学习算法选型
- **算法**：Soft Actor-Critic (SAC)。
- **决策理由**：相比 TD3/PPO，SAC 最大化熵的特性鼓励探索（Exploration），这在多智能体互动的动态环境中能有效防止陷入局部最优。
- **状态空间 (28维)**：24线激光雷达（降采样） + 2维相对目标坐标 + 2维自身速度。
- **动作空间 (2维)**：连续控制线速度 ($v$) 和角速度 ($\omega$)。

### 2.2 P2P 通信与聚合机制
- **拓扑结构**：动态 P2P 网状网络（无中心服务器）。
- **触发机制 (Event-Triggered)**：
  - **空间条件**：欧氏距离 $d < R_{comm}$ (模拟通信范围)。
  - **时间条件**：冷却时间 $t - t_{last} > T_{cooldown}$ (防止频繁抖动，节省带宽)。
- **聚合策略**：
  - 仅交换 **Actor 网络**参数（轻量级），Critic 网络本地保留（保护隐私且减少传输量）。
  - 采用 **Polyak Averaging** (软更新) 融合邻居权重：$\theta_{local} \leftarrow \beta \theta_{local} + (1-\beta) \theta_{neighbor}$。

### 2.3 安全防御体系 (创新点)
为了防御“投毒攻击”（恶意节点发送全零或高斯噪声参数），我实现了三种防御策略进行对照：
1. **Cosine Similarity (余弦相似度)**：计算本地模型与接收模型的向量夹角，拒绝相似度低于 $\mu - 3\sigma$ 的更新。
2. **Trimmed-Mean (截断平均)**：在聚合前移除参数分布中最大和最小的 $k\%$，消除极端异常值的影响。
3. **Krum 算法**：基于欧氏距离选择最“几何中心”的模型作为基准，理论上能防御 $f < n/2$ 个恶意节点。

### 2.4 工程化落地 (ROS2 & Python)
- **Sim2Real 架构**：设计了 `ROS2StateAdapter` 适配器模式，一套代码可同时运行在纯 Python 模拟器（加速训练）和 Gazebo/真实机器人（部署）上。
- **通信健壮性**：
  - **分片传输**：将大的神经网络权重 Tensor 切分为小块（Chunking），防止超过 UDP/DDS 单包限制。
  - **校验机制**：实现 CRC32 校验，丢弃损坏的数据包。
  - **可靠性**：实现了应用层重传机制，应对弱网环境。

## 3. 实验成果与量化指标 (Result)

我构建了完整的自动化实验流水线 (`run_full_pipeline.py`)，在 3 机器人协同导航任务中得出了以下结论：

1. **收敛速度**：
   - P2P 联邦学习相比独立训练（Local），收敛速度提升约 **40%**，达到相同奖励值所需的 Timesteps 显著减少。
   - 最终性能接近中心化联邦学习（Centralized），但去除了对中心服务器的依赖。

2. **通信开销**：
   - 相比全量参数交换，仅交换 Actor 网络使得单次通信量减少 **50%**。
   - 引入冷却门控机制（Cooldown Gate）后，总通信频次降低了 **60%**，且未显著影响收敛精度。

3. **鲁棒性 (防御效果)**：
   - 在 33% 节点为恶意投毒者（发送高斯噪声）的情况下：
     - **无防御**：群体模型迅速崩溃，奖励值跌至随机水平。
     - **Krum/Trimmed-Mean**：成功维持了 **85%+** 的原有性能，证明了系统的拜占庭容错能力。

## 4. 面试 Q&A 预演

**Q: 为什么只交换 Actor 不交换 Critic？**
A: 在 SAC 中，Critic 拟合的是本地的状态-动作价值（Q值），这与本地环境的特定 Replay Buffer 强相关。Actor 是策略函数，更具备通用性。只交换 Actor 既能实现策略知识共享，又能保护 Critic 对本地数据的隐私，同时显著降低带宽。

**Q: 你的系统如何处理网络延迟或丢包？**
A: 我采用了**异步更新**策略。机器人不需要等待邻居的响应（非阻塞），收到权重即刻融合。同时，我在应用层实现了分片重组和 CRC 校验，如果发生丢包导致校验失败，会直接丢弃本次更新，利用强化学习的容错性，等待下一次正常的通信，而不是卡死等待。

**Q: 为什么选择 ROS2？**
A: ROS2 基于 DDS（数据分发服务），天生支持去中心化的节点发现和发布/订阅模式，非常契合 P2P 联邦学习的拓扑需求。同时 ROS2 是工业界标准，方便将算法直接部署到 TurtleBot 等真实机器人上。

**Q: 实验中的难点是什么？**
A: 最大的难点是**Sim2Real的鸿沟**和**通信效率**。
- 为了解决训练慢的问题，我先编写了纯 Python 的轻量级模拟器进行算法验证（快 100 倍），验证通了再上 Gazebo。
- 为了解决大模型传输阻塞控制回路的问题，我实现了多线程收发和权重压缩。

## 5. 项目代码亮点

- **流水线自动化**：编写了 `run_full_pipeline.py`，一键完成从 Warm-up 到攻击防御对照实验的全流程，自动生成 TensorBoard 日志和对比图表。
- **模块化设计**：将 P2P 逻辑 (`p2p.py`) 与 RL 算法 (`sac.py`) 和环境 (`env.py`) 完全解耦，方便替换不同的防御策略或 RL 算法。
- **质量保证**：引入了 `unittest` 覆盖了核心通信编解码和防御算法，并使用配置快照 (`config_snapshot`) 确保实验可复现。
